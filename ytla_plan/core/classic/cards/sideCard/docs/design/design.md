# 侧边栏卡片

## 故事
侧边栏卡片是YTLA中最先设计出现的产物.  
说起来，侧边栏卡片的最初目的非常非常简单： 

摸鱼.  

当时还是2025年的春节，deepseek的出现突然把国内的LLM推火了.  
而那时，通过对deepseek的使用，我很快就意识到可以用它写自己想写的代码了：  
哪怕我不会那门语言，也可以硬上.  

所以，在公司部门里提出要开始做虚拟项目的最初期，  
我在听到风声之后，赶在虚拟项目正式开始前2个星期，就自费买了我现在的这台笔记本开发机，  
接上了自己的手机热点（我的这台笔记本始终都只接自己的热点），很快就冒出了第一个鬼主意：  

设计一个网页侧边栏，用作Plan manager & helper的目的，显示各种卡片.  

这个设计思路几乎是上手就确定的.  
第一版的侧边栏卡片功能，是flask + 原生js的形式.  

针对卡片本身的数据形式，其实确定得非常快. python-flask的基本功也还不错，这部分很快就完成了.  
难点在于前端. 最初是javascript原生实现，但是写得非常累.  
在那个时候，deepseek对于编程的水平似乎还是有限的，而原生js要实现一个简单的侧边栏的代码复杂度其实也非常高.  
所以很快，我就想到要更换前端语言框架了.  

我在vue和react上其实是做过衡量的.  
最后还是选择了vue.  
然后一个星期的时间，我把javascript的侧边栏替换成了vue3的样式，并且做出了非常新颖的显示效果.  
甚至直接用上了折叠卡片和masonry.  
这个时候，侧边栏卡片就已经有了定样.  

但后来我就又开始思考，这些卡片要怎么扩展，还能做什么.  
能不能实现出卡片即游戏？  
能不能实现出多元化卡片？  
能不能让卡片不单单长度可变，宽度也可变？  
这些内容最后一步步实现，就确定了现在的这套sideCard系统.  

# 设计

侧边栏卡片的设计是：  
一款基础的卡片 —> 一种类型的套卡 —> 一系列具体功能的卡片  
而type - subtype的这个规则就是在设计卡片的时候确定下来的.  

SideCard是最初的基类.
