# 分析：rule_harness_instructions.md 的非IDE依赖缺失内容

## 一、核心缺失分析

### 1. 工作模式的触发机制
- **原文档依赖**：通过 IDE 内置的 `/spec`、`/plan` 命令切换工作模式
- **非IDE缺失**：没有这些命令时，如何启动不同的工作阶段？

### 2. 工具集的可用性
原文档严重依赖以下IDE内置工具：
- `TodoWrite` - 任务管理
- `Read` / `Write` / `Edit` / `SearchReplace` - 文件操作
- `AskUserQuestion` - 用户交互
- `NotifyUser` - 通知用户
- `subagent` / `general_purpose_task` / `search` - 子代理调用
- `LS` / `Glob` / `Grep` - 代码库探索

### 3. 系统提示词和上下文
- **原文档依赖**：IDE提供的系统提示词（如 <system-reminder>）
- **非IDE缺失**：没有这些预定义的上下文和指令

### 4. 文件操作权限和路径
- **原文档依赖**：IDE内置的文件系统访问
- **非IDE缺失**：需要明确说明如何进行文件操作

### 5. 子代理的启动机制
- **原文档依赖**：IDE提供的子代理调用接口
- **非IDE缺失**：如何在没有IDE的情况下使用子代理协作？

## 二、需要调整的内容

### 1. 工作流程的启动方式
- 需要从"接收 /spec 命令"改为"用户明确要求开始规划"
- 需要明确说明不同阶段的切换条件

### 2. 工具替代方案
- 需要提供不依赖IDE工具的替代方法
- 文件操作：可以用标准的文本编辑方式描述
- 用户交互：可以用自然语言对话方式
- 子代理：可以用"请独立完成以下任务并返回结果"的方式

### 3. 系统上下文的补充
- 需要在文档开头明确说明当前的工作环境假设
- 需要补充必要的系统级指令

### 4. 文件路径的处理
- 需要明确说明相对于项目根目录的路径概念
- 需要提供目录结构的说明

### 5. 验证和反馈机制
- 需要调整验收标准的验证方式
- 需要明确用户确认的流程

## 三、保留的核心内容

以下内容不依赖IDE，可以保留：
- 整体的工作理念（规划先行、子代理协作、验证驱动等）
- 文档规范（origin.md、spec.md、tasks.md、checklist.md、actions.md 的格式）
- 质量检查清单
- 最佳实践
- 常见陷阱和避免方法
- 术语表

## 四、总结

要创建非IDE依赖的版本，需要：
1. 移除所有对IDE特定工具和命令的引用
2. 补充系统级的上下文和指令
3. 提供工具的替代方案
4. 调整工作流程的启动和切换方式
5. 明确文件操作和用户交互的方法
