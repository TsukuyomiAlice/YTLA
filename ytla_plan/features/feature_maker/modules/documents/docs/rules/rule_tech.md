# YTLA - Tech 文档规范

## 1. 总述

### 1.1 适用范围

- 适用核心名: 全部
- 适用栈: 包括前后端
- 适用语言类型和框架: 全部
- 指定规范目录：docs / documents（根据框架不同）
- 指定规范文档：tech 目录下的 .md 文档

### 1.2 设计理念

- **约定优于配置** - 通过标准化的目录结构和文件命名实现自动化
- **规划先行** - 先规划再执行，确保变更可控
- **子代理协作** - 合理使用子代理实现任务分解和并行处理
- **验证驱动** - 每个任务都有明确的验收标准
- **灵活适配** - 没有固定模板，根据实际模块内容灵活组织

## 2. Tech 文档概述

### 2.1 Tech 文档定义

Tech 文档是针对**模组/整个 feature 层面**的技术说明文档。它详细描述一个模块或特性的技术架构、实现细节、API 接口、数据流程、配置部署等技术层面的内容。

### 2.2 Tech 文档的用途

- 为开发者提供模块的技术全景图
- 记录架构设计决策和技术选型理由
- 说明 API 接口的使用方法和参数
- 描述数据处理流程和数据模型
- 提供配置部署和扩展开发的指导
- 便于新成员快速理解和上手模块

## 3. Tech 文档内容组织

### 3.1 没有固定模板

**重要说明：Tech 文档没有固定的标准模板！**

Tech 文档的内容应该根据被分析的模块或 feature 的实际情况来灵活组织。不同的模块可能有完全不同的结构和内容侧重点。

### 3.2 常见内容章节（参考）

虽然没有固定模板，但根据实际项目中的 Tech 文档，常见的内容章节包括（但不限于）：

#### 3.2.1 模块概述类
- 模块定位
- 主要功能
- 设计目标
- 在系统中的位置
- 使用场景

#### 3.2.2 结构说明类
- 文件结构说明
- 核心目录说明
- 关键文件说明
- 文件命名规范

#### 3.2.3 组件说明类
- 核心组件说明
- 组件职责和接口
- 组件关系图
- 设计模式说明

#### 3.2.4 API 接口类
- API 端点概览
- 详细接口说明
- 认证和授权
- 使用示例
- 错误码说明

#### 3.2.5 数据流程类
- 数据流程图
- 数据流向详细说明
- 核心数据处理场景
- 数据模型说明
- 错误处理和异常流程

#### 3.2.6 配置部署类
- 环境要求
- 配置选项
- 部署步骤
- 环境变量配置
- 常见问题排查
- 性能优化建议

#### 3.2.7 扩展开发类
- 扩展架构概览
- 添加新功能的步骤
- 扩展最佳实践
- 集成测试指南
- 常见扩展场景
- 代码贡献指南
- 故障排除

### 3.3 内容组织原则

1. **完整性** - 涵盖模块的主要技术方面
2. **逻辑性** - 章节之间有清晰的逻辑顺序
3. **实用性** - 内容对开发者有实际帮助
4. **可读性** - 使用清晰的标题、列表、代码示例等
5. **可维护性** - 便于后续更新和维护

## 4. Tech 文档维护工作流

### 4.1 Tech 文档创建流程

1. 定位目标模块/特性目录
2. 识别文档位置：
    - documents/tech/[模块名].md
    - docs/tech/[模块名].md
3. 全面探索模块内的代码实现
4. 理解模块的架构设计和技术要点
5. 根据模块特点规划文档结构
6. 编写文档内容
7. 添加代码示例、图表等辅助内容
8. 验证文档准确性
9. 更新作者名（如需要）
10. 更新文件更新日期

### 4.2 Tech 文档更新流程

1. 定位目标 Tech 文档
2. 分析代码变更内容
3. 更新文档中相应的章节
4. 确保文档与代码保持一致
5. 更新文件更新日期
6. 更新变更记录

### 4.3 作者名约定

- 一般不要主动改动
- 如果出现 "Official"，表明是官方文档，同步更新所有作者名为 "Official"
- 如果用户修改了作者名，同步修改所有相关文档

### 4.4 文件更新日期约定

- 格式：YYYY-MM-DD
- 只有在实际更新了内容时才更新日期
- 没有内容变更时不要更新日期

## 5. 文档目录结构规范

### 5.1 标准目录结构

```
原目录/
├── readme.md                    # 根目录 readme
└── documents/                   # 或 docs/（根据框架）
    ├── readme/
    ├── tech/                    # 技术文档目录
    │   ├── [模块1].md
    │   ├── [模块2].md
    │   └── ...
    ├── design/
    ├── implements/
    ├── rules/
    ├── discuss/
    └── interfaces/
```

### 5.2 文档命名规范

- 使用能反映文档内容的名称
- 通常使用模块名或特性名作为文件名
- 如：`sideCard.md`、`userAuth.md`、`database-connection.md`

## 6. Tech 文档检查清单

- [ ] 文档涵盖了模块的主要技术方面
- [ ] 文档结构清晰、逻辑合理
- [ ] 代码示例正确（如有）
- [ ] 图表清晰易懂（如有）
- [ ] API 接口说明完整（如有）
- [ ] 作者名正确
- [ ] 文件更新日期正确（如有变更）
- [ ] 文档内容与代码实现一致

## 7. Tech 文档维护最佳实践

- **同步更新** - 代码变更时同步更新 Tech 文档
- **准确反映** - 技术内容要准确反映当前实现
- **详细具体** - 核心技术实现部分要详细
- **代码示例** - 关键技术点可包含代码示例
- **灵活组织** - 根据模块特点灵活组织文档结构，不要拘泥于固定模板

## 8. Tech 文档维护常见陷阱和避免方法

❌ **陷阱**：使用固定模板，不考虑模块特点  
✅ **避免**：根据模块的实际内容灵活组织文档结构

❌ **陷阱**：Tech 文档与实际代码实现脱节  
✅ **避免**：代码变更后及时更新 Tech 文档

❌ **陷阱**：内容过于笼统，缺乏具体实现细节  
✅ **避免**：核心技术实现部分要详细说明关键技术点

❌ **陷阱**：没有内容变更也更新日期  
✅ **避免**：只有实际更新了内容时才更新日期
