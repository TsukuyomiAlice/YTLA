# YTLA - Implements 文档规范

## 1. 总述

### 1.1 适用范围

- 适用核心名: 全部
- 适用栈: 包括前后端
- 适用语言类型和框架: 全部
- 指定规范目录：docs / documents（根据框架不同）
- 指定规范文档：implements 目录下的 .md 文档

### 1.2 设计理念

- **约定优于配置** - 通过标准化的目录结构和文件命名实现自动化
- **规划先行** - 先规划再执行，确保变更可控
- **子代理协作** - 合理使用子代理实现任务分解和并行处理
- **验证驱动** - 每个任务都有明确的验收标准
- **单文档聚焦** - 针对单一文档进行深度技术说明
- **灵活适配** - 没有固定模板，根据实际文件内容灵活组织

## 2. Implements 文档概述

### 2.1 Implements 文档定义

Implements 文档是针对**单一文档/功能**的技术说明文档。它详细描述一个具体文件、脚本或功能模块的技术实现细节、API 接口、使用方法等技术层面的内容。

### 2.2 Implements 文档的用途

- 为开发者提供单个文件的技术细节说明
- 记录关键函数和 API 的使用方法
- 说明数据结构和输出格式
- 提供使用示例和依赖说明
- 便于理解和复用单个文件的功能

## 3. Implements 文档内容组织

### 3.1 没有固定模板

**重要说明：Implements 文档没有固定的标准模板！**

Implements 文档的内容应该根据被分析的单个文件或功能的实际情况来灵活组织。不同的文件可能有完全不同的结构和内容侧重点。

### 3.2 常见内容章节（参考）

虽然没有固定模板，但根据实际项目中的 Implements 文档，常见的内容章节包括（但不限于）：

#### 3.2.1 概述类
- Overview（概述）
- 功能介绍
- 主要特性
- 设计目标

#### 3.2.2 结构类
- Project Structure（项目结构）
- 文件结构说明
- 目录关系
- 依赖关系

#### 3.2.3 API 类
- Module API（模块 API）
- 主要入口点
- 核心函数说明
- 参数和返回值
- 使用示例

#### 3.2.4 实现类
- Core Functions（核心函数）
- 关键实现细节
- 算法说明
- 数据处理逻辑

#### 3.2.5 数据类
- Output Structure（输出结构）
- 数据模型说明
- JSON 输出格式
- 数据字段说明

#### 3.2.6 使用类
- Usage（使用方法）
- 作为模块使用
- 命令行使用
- 集成示例

#### 3.2.7 依赖类
- Dependencies（依赖）
- 外部库依赖
- 版本要求
- 安装说明

### 3.3 内容组织原则

1. **针对性强** - 只针对单一文件/功能进行说明
2. **实用性** - 内容对开发者有实际帮助
3. **示例丰富** - 提供充足的代码示例
4. **清晰明了** - 使用清晰的标题、列表、代码示例等
5. **可维护性** - 便于后续更新和维护

## 4. Implements 文档维护工作流

### 4.1 Implements 文档创建流程

1. 定位目标文档/文件位置
2. 识别对应的 Implements 文档位置：
    - documents/implements/[文档名].md
    - docs/implements/[文档名].md
3. 全面探索目标文件的内容和实现
4. 理解文件的功能、API 和技术要点
5. 根据文件特点规划文档结构
6. 编写文档内容
7. 添加代码示例、数据结构说明等
8. 验证文档准确性
9. 更新作者名（如需要）
10. 更新文件更新日期

### 4.2 Implements 文档更新流程

1. 定位目标 Implements 文档
2. 分析目标文件的变更内容
3. 更新文档中相应的章节
4. 确保文档与代码保持一致
5. 更新文件更新日期
6. 更新变更记录

### 4.3 作者名约定

- 一般不要主动改动
- 如果出现 "Official"，表明是官方文档，同步更新所有作者名为 "Official"
- 如果用户修改了作者名，同步修改所有 Implements 文档

### 4.4 文件更新日期约定

- 格式：YYYY-MM-DD
- 只有在实际更新了内容时才更新日期
- 没有内容变更时不要更新日期

## 5. 文档目录结构规范

### 5.1 标准目录结构

```
原目录/
├── docs/                   # 或 documents/（根据框架）
│   ├── implements/
│   │   ├── [文档1].md      # 针对文档1的技术说明
│   │   ├── [文档2].md      # 针对文档2的技术说明
│   │   └── [其它文档].md
│   ├── readme/
│   ├── tech/
│   ├── design/
│   ├── rules/
│   ├── discuss/
│   └── interfaces/
```

### 5.2 文档命名规范

- 使用能反映目标文档内容的名称
- 通常与目标文件同名（.md 后缀）
- 如：`scriptImportAnalyzer.md`、`dataProcessor.md`

## 6. Implements 文档检查清单

- [ ] 文档针对单一文件/功能
- [ ] 文档涵盖了文件的主要技术方面
- [ ] 文档结构清晰、逻辑合理
- [ ] 代码示例正确（如有）
- [ ] API 说明完整（如有）
- [ ] 作者名正确
- [ ] 文件更新日期正确（如有变更）
- [ ] 文档内容与代码实现一致

## 7. Implements 文档维护最佳实践

- **主语言优先** - 先更新主语言版本，再同步其他语言
- **针对性强** - 只针对单一文档进行技术说明，避免范围蔓延
- **详细具体** - 技术实现说明要详细，包含具体的实现细节
- **同步更新** - 当目标文档更新时，及时更新对应的 Implements 文档
- **灵活组织** - 根据文件特点灵活组织文档结构，不要拘泥于固定模板

## 8. Implements 文档维护常见陷阱和避免方法

❌ **陷阱**：参考其他文档的 implements.md  
✅ **避免**：根据当前目标文档的实际内容生成

❌ **陷阱**：使用固定模板，不考虑文件特点  
✅ **避免**：根据文件的实际内容灵活组织文档结构

❌ **陷阱**：随意修改作者名  
✅ **避免**：遵循作者名约定，除非用户明确要求

❌ **陷阱**：没有内容变更也更新日期  
✅ **避免**：只有实际更新了内容时才更新日期

❌ **陷阱**：范围过大，涉及多个文档  
✅ **避免**：保持单文档聚焦，一个 implements 文档只针对一个目标文档

❌ **陷阱**：技术说明过于笼统  
✅ **避免**：提供具体的实现细节、代码示例和使用说明

## 9. Implements 文档维护提示词

```
你正在维护这个应用的 Implements 文档。
用户应该告诉了你需要维护文档的目标文档路径。
这个应用的 Implements 文档目录位于 docs / documents/implements 文件夹下。

你需要做的是，结合目标文档的内容，创建或更新对应的 Implements 文档：
- 文档命名：与目标文档同名（.md 后缀）
- 存放位置：docs/documents/implements/ 目录下

重要提示：Implements 文档没有固定的标准模板！
你需要根据被分析的单个文件或功能的实际情况来灵活组织文档内容。

Implements 文档用于针对单一文档的技术说明。
你应该先完整地查看目标文档的内容，理解文件的功能、API 和技术要点。

根据文件特点，Implements 文档可能包含（但不限于）以下内容：
- Overview（概述）
- Project Structure（项目结构）
- Key Features（主要特性）
- Module API（模块 API）
- Core Functions（核心函数）
- Output Structure（输出结构）
- Usage（使用方法）
- Dependencies（依赖）

请根据文件的实际情况，选择合适的章节来组织文档。

并不建议你参考其它的 Implements 文档来为目标文档更新。
实际上，Implements 文档是没有固定模板的。
你只需要根据目标文件的实际内容灵活组织即可。

**作者名** 有一个特别约定：
一般不要主动改动。
但是，如果出现了 "Official"，则表明这个文档是官方文档。你要同步更新所有的作者名为 "Official"
如果用户修改了其它的作者名，那么做同步的修改，把所有的 Implements 文档中的作者名改成相同的名字。

如果发生了更新的话，顺带更新一下文档里的**文件更新日期**。格式为 "YYYY-MM-DD"。
但是，如果没有更新任何内容，那么不要更新文件更新日期。

有时候，用户可能会在其它语言目录中更新了内容，而没有更新主目录的 Implements 文档。
这时候你需要进行比对，确认哪一个版本才是最具体的版本，并以那个版本所使用的语言作为主语言，更新并同步所有的 Implements 文档。
```
