# YTLA - Docs 文件规范（Harness Engineering 版本）

## 1. 总述

### 1.1 适用范围
- 适用核心名: 全部  
- 适用栈: 包括前后端  
- 适用语言类型和框架: 全部
- 指定规范目录：docs / documents（根据框架不同）
- 指定规范文档：readme.md

### 1.2 设计理念
- **约定优于配置** - 通过标准化的目录结构和文件命名实现自动化
- **规划先行** - 先规划再执行，确保变更可控
- **子代理协作** - 合理使用子代理实现任务分解和并行处理
- **验证驱动** - 每个任务都有明确的验收标准

## 2. 工作模式

### 2.1 Spec Mode（规划模式）
**适用场景**：需要详细规划的任务，如新增功能、重构、文档维护等

**工作流程**：
1. 分析当前代码库状态
2. 生成 PRD (spec.md) - 产品需求文档
3. 生成 Implementation Plan (tasks.md) - 实现计划
4. 生成 Verification Checklist (checklist.md) - 验证清单
5. 通知用户审查规划文档
6. 等待用户批准后进入 Agent Mode

### 2.2 Agent Mode（执行模式）
**适用场景**：已规划好的任务，或简单的单步任务

**工作流程**：
1. 选择优先级最高的待处理任务
2. 标记任务为进行中
3. 如需复杂任务，委派给子代理
4. 执行任务并进行验证
5. 更新任务状态
6. 重复直到所有任务完成

## 3. 规划文档规范

### 3.1 spec.md (Product Requirements Document) 规范

**必填章节**：
- Overview - 概要
- Goals - 目标
- Non-Goals - 非目标（边界）
- Background & Context - 背景和上下文
- Functional Requirements - 功能需求
- Non-Functional Requirements - 非功能需求
- Constraints - 约束条件
- Assumptions - 假设
- Acceptance Criteria - 验收标准（Given/When/Then 格式）
- Open Questions - 待解决问题

**验收标准格式要求**：
```markdown
### AC-1: [标准名称]
- **Given**: [前置条件]
- **When**: [触发动作]
- **Then**: [可观察的结果]
- **Verification**: `programmatic` | `human-judgment`
- **Notes**: [补充说明]
```

### 3.2 tasks.md (Implementation Plan) 规范

**任务格式要求**：
```markdown
## [ ] Task [序号]: [任务标题]
- **Priority**: P0 | P1 | P2
- **Depends On**: [依赖的任务 ID 或 "None"]
- **Description**: 
  - [需要实现的内容]
  - [关键实现细节]
- **Acceptance Criteria Addressed**: [AC-1, AC-2, ...]
- **Test Requirements**:
  - `programmatic` TR-[序号].1: [可测量的验证点]
  - `human-judgement` TR-[序号].2: [人工审查的要点]
- **Notes**: [边界情况、风险、实现提示]
```

**优先级定义**：
- **P0** - 关键路径，阻塞其他工作，核心功能
- **P1** - 重要但不阻塞，次要功能
- **P2** - 锦上添花，优化，完善

### 3.3 checklist.md (Verification Checklist) 规范

**格式要求**：
```markdown
- [ ] Checkpoint 1: [简洁具体的验证检查点描述]
- [ ] Checkpoint 2: [简洁具体的验证检查点描述]
...
```

**注意事项**：
- 每个检查点应该是可验证的
- 避免模糊的描述
- 按逻辑顺序排列

## 4. 文档维护工作流

### 4.1 完整工作流

```
1. 用户发起请求
   ↓
2. 分析当前状态（探索代码库）
   ↓
3. 进入 Spec Mode（如需要）
   ├─ 生成 spec.md
   ├─ 生成 tasks.md
   ├─ 生成 checklist.md
   └─ 通知用户审查
   ↓
4. 用户批准规划
   ↓
5. 进入 Agent Mode
   ├─ 选择 P0 任务
   ├─ 委派给子代理（如需要）
   ├─ 执行任务
   ├─ 验证任务
   ├─ 更新任务状态
   └─ 重复直到完成
   ↓
6. 最终验证（运行 checklist）
   ↓
7. 完成并总结
```

### 4.2 Readme 文档维护流程

**步骤**：
1. 定位目标应用目录
2. 识别文档位置：
   - 根目录 readme.md
   - documents/readme/zh-CN/readme.md
   - documents/readme/en-US/readme.md
3. 探索应用内的代码文件结构
4. 确定主语言（通常根目录 readme.md 的语言）
5. 填充占位符内容：
   - 概念部分：简明描述应用用途
   - 文件结构部分：列出完整文件列表
6. 同步多语言版本
7. 更新作者名（如需要）
8. 更新文件更新日期（如有内容变更）

### 4.3 多语言同步规范

**规则**：
- 优先更新根目录 readme.md
- 根据根目录语言确定主语言
- 生成其他语言的翻译版本
- 确保三个文档内容一致，仅语言不同
- 如发现其他语言版本有更新但主版本未更新，需要比对并同步

## 5. 子代理使用规范

### 5.1 何时使用子代理

**推荐使用场景**：
- 任务可以分解为多个独立的子任务
- 任务需要大量的上下文探索（可能产生很多不需要的 token）
- 任务可以并行处理
- 任务是黑盒验证（只需要结果，不需要过程）

**不推荐使用场景**：
- 单步简单任务
- 任务间有紧密的依赖关系
- 需要保留完整的上下文用于后续任务

### 5.2 子代理委派规范

**委派任务时的提示词要点**：
- 清晰的任务描述
- 明确的验收标准
- 提供必要的上下文
- 指定返回格式
- 说明如何验证

**示例**：
```
描述：[3-5 个词的简短描述]
任务：[详细的任务说明，<= 30 个词]
返回语言：zh-CN
```

### 5.3 子代理协作模式

**串行模式**：
```
主代理 → 子代理 1 → 子代理 2 → ... → 完成
```
适用：任务间有依赖关系

**并行模式**：
```
         ┌→ 子代理 1 ─┐
主代理 ──┼→ 子代理 2 ─┼→ 汇总 → 完成
         └→ 子代理 3 ─┘
```
适用：任务间相互独立

## 6. Readme 文档内容规范

### 6.1 标准模板

```markdown
<p>
  语言
 <a href="./documents/readme/zh-CN/readme.md"> 简体中文 </a>
 <a href="./documents/readme/en-US/readme.md"> English </a>
</p>


# [模块名]

### [YTLA应用 / YTLA核心应用 / YTLA特性 / YTLA核心特性 / YTLA应用 公共配置 / YTLA核心应用 公共配置]

### [作者名]

version [核心名]

[前端/后端]语言及开发框架: [技术栈]  
适用YTLA core版本: **[core版本]**  
文件更新日期: [YYYY-MM-DD]

## 概念

[简明描述这个应用用来做什么]

## 特性包目录

[列出应用下的完整文件列表]
```

### 6.2 作者名约定

- 一般不要主动改动
- 如果出现 "Official"，表明是官方应用，同步更新所有作者名为 "Official"
- 如果用户修改了作者名，同步修改所有 readme 文档

### 6.3 文件更新日期约定

- 格式：YYYY-MM-DD
- 只有在实际更新了内容时才更新日期
- 没有内容变更时不要更新日期

### 6.4 概念部分规范

- 简明描述应用用途
- 1-3 句话为宜
- 避免技术细节，保持概念性

### 6.5 文件结构部分规范

- 列出完整的文件和目录
- 使用缩进表示层级
- 准确反映当前实际文件结构
- 按字母顺序或逻辑顺序排列

## 7. 文档目录结构规范

### 7.1 标准目录结构

```
原目录/
├── readme.md                    # 根目录 readme（主语言）
└── documents/                   # 或 docs/（根据框架）
    ├── readme/
    │   ├── zh-CN/
    │   │   └── readme.md       # 简体中文版
    │   ├── en-US/
    │   │   └── readme.md       # 英文版
    │   └── [其它语种]/
    ├── tech/                    # 技术文档
    ├── design/                  # 设计文档
    ├── implements/              # 实现文档
    ├── rules/                   # 规范文档
    ├── discuss/                 # 讨论文档
    └── interfaces/              # 接口文档
```

### 7.2 文档类型说明

| 目录名        | 用途                                  |
|------------|-------------------------------------|
| readme     | 文档 readme 目录（必须包含）          |
| tech       | 针对模组/整个 feature 层面的技术说明     |
| design     | 介绍想法设计, 功能 story, 用户 scenario 等 |
| implements | 针对单一文档的技术说明                 |
| rules      | 规范                                  |
| discuss    | 设计讨论, 进度汇总, todo 清单 等文档      |
| interfaces | 接口设计文档，如 openAPI 的 yaml 文档等    |

### 7.3 目录名称适配

- Python-Flask 框架：使用 `docs/`
- Vue 3 框架：使用 `documents/`
- 其他框架：根据实际情况调整

## 9. 最佳实践

### 9.1 规划最佳实践

- **先探索后规划** - 充分理解当前状态再做规划
- **边界清晰** - 明确什么是 IN，什么是 OUT
- **可验证的标准** - 每个验收标准都应该是可验证的
- **合理分解** - 任务粒度要适中，既不过大也不过小
- **优先级正确** - 先做关键路径上的任务

### 9.2 执行最佳实践

- **一次一个** - 一次只执行一个任务
- **及时更新** - 任务状态要及时更新
- **合理委派** - 复杂任务委派给子代理
- **验证驱动** - 完成任务后立即验证
- **渐进式** - 完成一个任务再进入下一个

### 9.3 文档维护最佳实践

- **主语言优先** - 先更新主语言版本，再同步其他语言
- **准确反映** - 文件结构要准确反映当前实际状态
- **简明扼要** - 概念部分要简明，避免冗长
- **同步更新** - 多语言版本要同步更新
- **日期准确** - 只有内容变更时才更新日期

## 10. 常见陷阱和避免方法

### 10.1 规划阶段陷阱

❌ **陷阱**：规划过于详细，导致执行时发现不可行  
✅ **避免**：保持规划在合理的抽象层次，留出调整空间

❌ **陷阱**：没有明确 Non-Goals，导致范围蔓延  
✅ **避免**：明确说明什么不做，设定清晰的边界

❌ **陷阱**：验收标准模糊，无法验证  
✅ **避免**：使用 Given/When/Then 格式，明确验证类型

### 10.2 执行阶段陷阱

❌ **陷阱**：同时执行多个任务，导致混乱  
✅ **避免**：一次只执行一个任务，完成一个再做下一个

❌ **陷阱**：不使用子代理，导致主代理上下文过大  
✅ **避免**：合理使用子代理分解任务

❌ **陷阱**：任务完成后不验证，导致后期发现问题  
✅ **避免**：每个任务完成后立即验证

### 10.3 文档维护陷阱

❌ **陷阱**：参考其他应用的 readme.md  
✅ **避免**：根据当前应用的实际文件结构生成

❌ **陷阱**：随意修改作者名  
✅ **避免**：遵循作者名约定，除非用户明确要求

❌ **陷阱**：没有内容变更也更新日期  
✅ **避免**：只有实际更新了内容时才更新日期

## 11. 提示词参考库

### 11.1 Readme 维护提示词

```
你正在维护这个应用的文档。
用户应该告诉了你需要维护文档的应用目录。
这个应用的文档目录位于 docs / documents 文件夹下。
另外，还有一份在应用根目录下的 readme.md 文档

你需要做的是，结合应用内存在的文件，更新以下范围的文档：
- 根目录的 readme.md
- 文档目录内的：readme 下的 .md 文档

readme 用于介绍应用的**概念**和**文件结构**。
**概念** 里简明描述这个应用用来做什么。
**文件结构** 需要列出应用下的文件列表。

一般而言，如果用户没有提出要求，你需要维护的是根目录的 readme.md，以及 readme 目录下的多语言版本文件。
你要根据根目录的 readme.md 的语言，确定文档的主语言，并生成其它文档的翻译版本。
你要先完整地查看应用内的代码文件，确定并替换生成 readme.md 中需要生成或更新的内容。
随后，同步其它的翻译版本。

并不建议你参考其它的应用或功能下的 readme.md 来为目标应用更新。
实际上，readme.md 是有预生成的统一格式的。
你只需要完成占位符的填充即可。

**作者名** 有一个特别约定：
一般不要主动改动。
但是，如果出现了 "Official"，则表明这个应用是官方应用。你要同步更新所有的作者名为 "Official"
如果用户修改了其它的作者名，那么做同步的修改，把所有的 readme 文档中的作者名改成相同的名字。

如果发生了更新的话，顺带更新一下文档里的**文件更新日期**。格式为 "YYYY-MM-DD"。
但是，如果没有更新任何内容，那么不要更新文件更新日期。

对于文档目录内其它目录下的文档，除非用户明确要求，一般不主动维护。

有时候，用户可能会在其它语言目录中更新了内容，而没有更新主目录的 readme.md。
这时候你需要进行比对，确认哪一个版本才是最具体的版本，并以那个版本所使用的语言作为主语言，更新并同步所有的 readme.md。
```

1. 研究当前代码库状态，理解用户需求
2. 生成 PRD (spec.md) - 产品需求文档
3. 生成 Implementation Plan (tasks.md) - 实现计划
4. 生成 Verification Checklist (checklist.md) - 验证清单
5. 通知用户审查规划文档

请使用中文编写所有规划文档。
```

```
