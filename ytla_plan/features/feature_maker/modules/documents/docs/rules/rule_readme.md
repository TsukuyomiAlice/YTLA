# YTLA - Readme 文档规范

## 1. 总述

### 1.1 适用范围

- 适用核心名: 全部
- 适用栈: 包括前后端
- 适用语言类型和框架: 全部
- 指定规范目录：docs / documents（根据框架不同）
- 指定规范文档：readme.md

### 1.2 设计理念

- **约定优于配置** - 通过标准化的目录结构和文件命名实现自动化
- **规划先行** - 先规划再执行，确保变更可控
- **子代理协作** - 合理使用子代理实现任务分解和并行处理
- **验证驱动** - 每个任务都有明确的验收标准

## 2. 文档维护工作流

### 2.1 Readme 文档维护流程

**步骤**：

1. 定位目标应用目录
2. 识别文档位置：
    - 根目录 readme.md
    - documents/readme/zh-CN/readme.md
    - documents/readme/en-US/readme.md
3. 探索应用内的代码文件结构
4. 确定主语言（通常根目录 readme.md 的语言）
5. 填充占位符内容：
    - 概念部分：简明描述应用用途
    - 文件结构部分：列出完整文件列表
6. 同步多语言版本
7. 更新作者名（如需要）
8. 更新文件更新日期（如有内容变更）
9. 更新变更记录(需要注意，如果这部分缺失，需要参照标准模板补齐，并以这次变更记录的内容为准添加)

### 2.2 多语言同步规范

**规则**：

- 优先更新根目录 readme.md
- 根据根目录语言确定主语言
- 生成其他语言的翻译版本
- 确保三个文档内容一致，仅语言不同
- 如发现其他语言版本有更新但主版本未更新，需要比对并同步

## 3. Readme 文档内容规范

### 3.1 脚手架自动生成

readme.md 文档可通过 scaffold(脚手架)特性自动生成初始模板。在创建新应用或特性时，脚手架会预生成符合标准格式的 readme.md
文件，包含基本的结构和占位符，开发者只需填充具体内容即可。

### 3.2 标准模板

```markdown
<p>
  语言
 <a href="./documents/readme/zh-CN/readme.md"> 简体中文 </a>
 <a href="./documents/readme/en-US/readme.md"> English </a>
</p>

# [模块名]

### [YTLA应用 / YTLA核心应用 / YTLA特性 / YTLA核心特性 / YTLA应用 公共配置 / YTLA核心应用 公共配置]

### [作者名]

version [核心名]

[前端/后端]语言及开发框架: [技术栈]  
适用YTLA core版本: **[core版本]**  
文件更新日期: [YYYY-MM-DD]

## 概念

[简明描述这个应用用来做什么]

## 特性包目录

[列出应用下的完整文件列表]

## 变更记录

[按以下格式添加更新记录，添加在本段落的最上方，不要删除过往内容:
yyyy-mm-dd
更新内容
]
```

### 3.3 作者名约定

- 一般不要主动改动
- 如果出现 "Official"，表明是官方应用，同步更新所有作者名为 "Official"
- 如果用户修改了作者名，同步修改所有 readme 文档

### 3.4 文件更新日期约定

- 格式：YYYY-MM-DD
- 只有在实际更新了内容时才更新日期
- 没有内容变更时不要更新日期

### 3.5 概念部分规范

- 简明描述应用用途
- 1-3 句话为宜
- 避免技术细节，保持概念性

### 3.6 文件结构部分规范

- 列出完整的文件和目录
  - 特别地, 在documents/tasks中的部分，只需要列出PRD目录列表就可以了。里面会重复大量的origin.md/spec.md/task.md/checklist.md则不需要
  - 在列出tasks目录时，不要忘记把最新生成的PRD目录也列出来
- 使用缩进表示层级
- 准确反映当前实际文件结构
- 按字母顺序或逻辑顺序排列

### 3.7 变更记录部分规范

- 按以下格式添加在本部分的最上方

```markdown
### YYYY-MM-DD (与文件更新日期相同)

[本次变更涉及的内容]
```

- 变更内容应简明描述
- 每个变更点1句话
- 如果文档中此部分缺失，应当添加此部分，且仅添加本次变更涉及的内容

## 4. 文档目录结构规范

### 4.1 标准目录结构

```
原目录/
├── readme.md                    # 根目录 readme（主语言）
└── documents/                   # 或 docs/（根据框架）
    ├── readme/
    │   ├── zh-CN/
    │   │   └── readme.md       # 简体中文版
    │   ├── en-US/
    │   │   └── readme.md       # 英文版
    │   └── [其它语种]/
    ├── tech/                    # 技术文档
    ├── design/                  # 设计文档
    ├── implements/              # 实现文档
    ├── rules/                   # 规范文档
    ├── discuss/                 # 讨论文档
    └── interfaces/              # 接口文档
```

### 4.2 文档类型说明

| 目录名        | 用途                              |
|------------|---------------------------------|
| readme     | 文档 readme 目录（必须包含）              |
| tech       | 针对模组/整个 feature 层面的技术说明         |
| design     | 介绍想法设计, 功能 story, 用户 scenario 等 |
| implements | 针对单一文档的技术说明                     |
| rules      | 规范                              |
| discuss    | 设计讨论, 进度汇总, todo 清单 等文档         |
| interfaces | 接口设计文档，如 openAPI 的 yaml 文档等     |
| tasks      | 由AI harness 自动生成的PRD文档          |

### 4.3 目录名称适配

- Python-Flask 框架：使用 `docs/`
- Vue 3 框架：使用 `documents/`
- 其他框架：根据实际情况调整

## 5. Readme 文档检查清单

- [ ] 概念部分清晰准确
- [ ] 文件结构部分完整
- [ ] 作者名正确
- [ ] 文件更新日期正确（如有变更）
- [ ] 多语言版本内容一致
- [ ] 链接正确
- [ ] 格式符合标准模板

## 6. Readme 维护最佳实践

- **主语言优先** - 先更新主语言版本，再同步其他语言
- **准确反映** - 文件结构要准确反映当前实际状态
- **简明扼要** - 概念部分要简明，避免冗长
- **同步更新** - 多语言版本要同步更新
- **日期准确** - 只有内容变更时才更新日期

## 7. Readme 维护常见陷阱和避免方法

❌ **陷阱**：参考其他应用的 readme.md  
✅ **避免**：根据当前应用的实际文件结构生成

❌ **陷阱**：随意修改作者名  
✅ **避免**：遵循作者名约定，除非用户明确要求

❌ **陷阱**：没有内容变更也更新日期  
✅ **避免**：只有实际更新了内容时才更新日期

## 8. Readme 维护提示词

```
你正在维护这个应用的文档。
用户应该告诉了你需要维护文档的应用目录。
这个应用的文档目录位于 docs / documents 文件夹下。
另外，还有一份在应用根目录下的 readme.md 文档

你需要做的是，结合应用内存在的文件，更新以下范围的文档：
- 根目录的 readme.md
- 文档目录内的：readme 下的 .md 文档

readme 用于介绍应用的**概念**和**文件结构**。
**概念** 里简明描述这个应用用来做什么。
**文件结构** 需要列出应用下的文件列表。

一般而言，如果用户没有提出要求，你需要维护的是根目录的 readme.md，以及 readme 目录下的多语言版本文件。
你要根据根目录的 readme.md 的语言，确定文档的主语言，并生成其它文档的翻译版本。
你要先完整地查看应用内的代码文件，确定并替换生成 readme.md 中需要生成或更新的内容。
随后，同步其它的翻译版本。

并不建议你参考其它的应用或功能下的 readme.md 来为目标应用更新。
实际上，readme.md 是有预生成的统一格式的。
你只需要完成占位符的填充即可。

**作者名** 有一个特别约定：
一般不要主动改动。
但是，如果出现了 "Official"，则表明这个应用是官方应用。你要同步更新所有的作者名为 "Official"
如果用户修改了其它的作者名，那么做同步的修改，把所有的 readme 文档中的作者名改成相同的名字。

如果发生了更新的话，顺带更新一下文档里的**文件更新日期**。格式为 "YYYY-MM-DD"。
但是，如果没有更新任何内容，那么不要更新文件更新日期。

对于文档目录内其它目录下的文档，除非用户明确要求，一般不主动维护。

有时候，用户可能会在其它语言目录中更新了内容，而没有更新主目录的 readme.md。
这时候你需要进行比对，确认哪一个版本才是最具体的版本，并以那个版本所使用的语言作为主语言，更新并同步所有的 readme.md。
```
